## The five confirmed loss paths

| # | Path | Effect |
|---|------|--------|
| 1 | Periodic tick never persists; saves only on discrete transport events | Force quit while playing → position reverts to last event (usually 0 = track start). **The reported bug.** |
| 2 | `updateNowPlaying()` with nil `currentTrack` calls `PlaybackStateStore.clear()` | Cold-launch Siri "resume", lock-screen command, deep link, or interruption-pause before restore completes → queue AND position permanently erased. |
| 3 | Restore persists elapsed=0 (via `loadCurrent` → `updateNowPlaying`) before seeking to `saved.elapsed`; seek uses default tolerance and can silently fail on unready/remote items | Kill during the window, or failed seek, → position lost. Also: any tick/persist after a failed seek overwrites the good value with ~0. |
| 4 | Only durable copy is App Group UserDefaults + rowid identity | Uninstall deletes it; after reinstall + CloudKit resync rowids differ, so even a durable copy keyed by rowid would restore the wrong/missing tracks. |
| 5 | `performQueueRestore` leaves `startIndex = 0` when the saved current track can't be resolved, but still seeks to `saved.elapsed` | Deleting the in-progress track applies the old position to an unrelated track. |

Additional hardening gaps: `currentTime` can be NaN before an item is ready →
`JSONEncoder` throws on NaN → `try?` silently skips the save (v1 keeps stale data —
OK — but sanitize anyway); no scenePhase flush; UserDefaults writes go through
cfprefsd (async) — fine in practice for force quit, but an atomic file is strictly
stronger and is what the file tier should read/write.

## Guarantee model (what "never lose, mathematically" means here)

iOS provides no callback on force quit, so zero-loss is achievable only by bounding
the persistence interval. The guarantee we implement and PROVE by model-based tests:

- **G1 (exactness on events):** after any discrete transport event (play, pause,
  seek, next/previous, track change, queue edit, interruption, route change, app
  backgrounding), the durable position equals the in-memory position exactly.
- **G2 (bounded loss while playing):** during steady playback the durable position is
  never more than **τ = 1.0 s** behind the true position (persist on tick, throttled
  to ≥1 write/s; ticks arrive every 0.5 s).
- **G3 (no regression):** a durable snapshot is never overwritten by one that moves
  the same track's position backwards, except as the direct result of an explicit
  user/transport action (seek back, previous, repeat loop, new queue).
- **G4 (no spurious erasure):** the durable snapshot is cleared ONLY by the explicit
  queue-clear path. No control-surface call, restore, or launch ordering can erase it.
- **G5 (atomicity):** every durable write is all-or-nothing (temp file + atomic
  rename). A torn/corrupt file never loses the last-known-good state.
- **G6 (reinstall):** position (queue syncIDs + index + elapsed + savedAt) is mirrored
  to CloudKit (throttled: every 30 s while playing + every discrete event), so after
  uninstall/reinstall the position is recovered to within the cloud throttle bound;
  identity is by `syncID`, immune to rowid changes.
- **G7 (restore correctness):** restore never writes a snapshot worse than the one it
  read (persistence is suppressed until the restore seek is confirmed), retries the
  seek until the player lands within 2 s of the target, and zeroes elapsed only when
  the saved current track no longer exists.

# Phase 2 — fixes

## F1. Snapshot v2 + atomic file tier (`Sources/Snapshot/`)

- Extend `PlaybackStateSnapshot`: `var trackSyncIDs: [String?]? = nil` (parallel to
  `trackIDs`). Optional → old JSON decodes unchanged. Sanitize on init/save:
  `elapsed = elapsed.isFinite ? max(0, elapsed) : 0`.
- New `PlaybackStateFileStore`: location = app-group container
  (`FileManager.containerURL(forSecurityApplicationGroupIdentifier:
  WidgetSnapshotStore.appGroupIdentifier)`) falling back to Application Support on

  `load` = decode; on decode failure return nil (never delete the file on failure —
  a later good write replaces it).
- Keep writing the v1 UserDefaults key on every save (widget/back-compat), but reads
  prefer the file tier: `load()` = max-by-`savedAt` of {file, defaults}.

## F2. Write admission policy (pure, host-tested)

New `Sources/Snapshot/PlaybackWritePolicy.swift`:

```swift
public enum PlaybackWriteReason {
    case tick, transportEvent, userSeek, queueChange, restoreCommit, background, userClear
}
public enum PlaybackWritePolicy {
    /// candidate == nil means "clear".
    public static func admits(candidate: PlaybackStateSnapshot?,
                              existing: PlaybackStateSnapshot?,
                              reason: PlaybackWriteReason) -> Bool
}
```

Rules (encode G3/G4 exactly):
- `candidate == nil` → admitted iff `reason == .userClear`.
- If `existing` describes the same queue (`trackIDs ==`) and same `currentIndex`,
  a candidate with `elapsed < existing.elapsed - 1.0` is admitted only for reasons
  `{userSeek, transportEvent, queueChange, restoreCommit}` — i.e. **`.tick` and
  `.background` may never move the position backwards** (kills the "transient 0 tick
  after item swap / failed restore seek" class of loss).
- Different queue or index → always admit (except nil rule above).

## F3. Persist on tick (fixes loss #1)

In `addPeriodicObserver()`'s closure, after `updateNowPlayingTime()`:
`persistTick()` — persists with reason `.tick` when `isAdvancing` and ≥1.0 s since
the last durable write (track `lastPersistAt`). Route ALL persistence through one
funnel: `persist(reason:)` → build snapshot → `PlaybackWritePolicy.admits` →
composite store save. `seek(to:)` uses `.userSeek`; `updateNowPlaying()` uses
`.transportEvent`; `clearQueuePlayback()` uses `.userClear`.

## F4. Never clear on nil currentTrack (fixes loss #2)

`updateNowPlaying()` nil branch: call ONLY `bridge.clearNowPlaying()`. Delete the
`PlaybackStateStore.clear()` there. The only clear left is `clearQueuePlayback()`.

## F5. Restore hardening (fixes loss #3 and #5)

- Extract pure `QueueRestorePlanner` (new `Sources/Audio/QueueRestorePlanner.swift`):
  `plan(saved: PlaybackStateSnapshot, resolveByID:, resolveBySyncID:) async ->
  RestorePlan { rows: [TrackRow], startIndex: Int, seekTo: Double }`. Semantics:
  resolve each entry by rowid, fall back to syncID; `seekTo = saved.elapsed` only if
  the saved CURRENT entry resolved, else 0 (loss #5); startIndex remapped over
  resolved rows (existing logic, but now testable).
- In `AudioPlayer`: add `private var isRestoring = false` — `persist(reason:)`
  no-ops while true (suppression beats admission here because `loadCurrent`'s write
  is a `.transportEvent`, which is allowed to regress).
- Restore sequence: `isRestoring = true` → apply plan (queue/index) →
  `loadCurrent(autoplay: false)` → set `currentTime = plan.seekTo` immediately →
  `player.seek(to:, toleranceBefore: .zero, toleranceAfter: .zero)` → set
  `pendingRestoreSeek = plan.seekTo` → `isRestoring = false` →
  `persist(reason: .restoreCommit)`.
- Seek confirmation: in the periodic tick, while `pendingRestoreSeek != nil`: if the
  item is `.readyToPlay` and `|currentTime - target| > 2` re-issue the exact seek;
  when within 2 s (or on any user seek/track change) clear `pendingRestoreSeek`.
  While pending, ticks report `currentTime = target` for persistence purposes (the
  admission policy additionally blocks tick regressions — belt and suspenders).
- Restore retry: `restorePersistedQueue()` may run again if the previous attempt
  restored nothing (`performQueueRestore` returned with queue still empty) — reset
  `restoreTask` in that case. Needed for the post-CloudKit-reconcile second attempt
  (F8) and makes Seam S3 real logic.
